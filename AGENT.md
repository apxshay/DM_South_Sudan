## Project Overview

This repository contains a university project for a Master's Degree course in Computer Engineering and Artificial Intelligence, focused on Data Management.

The main objective is to compare a traditional Relational Database Management System (RDBMS) with a Graph Database using the same real-world dataset.

The selected case study concerns humanitarian and infrastructure data from **South Sudan**, including:

* Road networks
* Hospitals
* Displacement sites / refugee camps

**GitHub repository:** [github.com/apxshay/DM_South_Sudan](https://github.com/apxshay/DM_South_Sudan)

The project will involve:

1. Data acquisition and exploration
2. Data cleaning and preprocessing
3. Data modeling
4. Relational database implementation
5. Graph database implementation
6. Query design
7. Transaction management
8. Benchmarking and performance evaluation
9. Comparative analysis and reporting

The final goal is not only to build two databases, but also to understand and demonstrate the strengths and weaknesses of relational and graph-based approaches when working with interconnected infrastructure data.

## Project Methodology

The project will be developed incrementally.

Before implementing anything, data must be understood.

The expected workflow is:

### Phase 1 — Data Understanding

Analyze all datasets and produce:

* Dataset descriptions
* Schemas
* Data types
* Missing values analysis
* Uniqueness and identifier analysis
* Cross-dataset observations
* Data quality issues

### Phase 2 — Data Modeling

Design:

* Relational schema
* Graph schema

The two models should represent the same domain while remaining natural for their respective database technologies.

### Phase 3 — Data Integration

Build ETL pipelines that:

* Load raw datasets
* Clean data
* Normalize formats
* Generate import-ready outputs

### Phase 4 — Database Population

Populate:

* PostgreSQL (or equivalent RDBMS)
* Neo4j (or equivalent Graph Database)

### Phase 5 — Query Design

Create equivalent queries for both systems.

Examples:

* Nearest hospital to a displacement site
* Reachability analysis
* Route discovery
* Infrastructure impact analysis
* Shortest path computation

### Phase 6 — Benchmarking

Compare:

* Insert performance
* Update performance
* Query execution time
* Complexity of implementation
* Expressiveness of query language

### Phase 7 — Final Evaluation

Produce a rigorous comparison between the two database paradigms.

### Datasets acquired

| Domain | Source | Local path | Records (key figure) |
|--------|--------|------------|----------------------|
| Roads (humanitarian) | [HDX — South Sudan Road Network](https://data.humdata.org/dataset/south-sudan-road-network_hdx) | `data/raw/roads/` | 976 line features (roads + rivers) |
| Roads (HOT OSM) | [HDX — Roads of South Sudan](https://data.humdata.org/dataset/hotosm_ssd_roads) | `data/raw/roads_hotosm/` | 20,512 filtered lines (`primary`, `secondary`, `tertiary`, `unclassified`) |
| Health facilities | [HDX — Health Facilities](https://data.humdata.org/dataset/south-sudan-health-facilities) | `data/raw/health_facilities/` | WHO 2025: 1,988 rows; SS 2023: 1,513 rows (75 hospitals) |
| IDP displacements | [HDX — IDMC IDPs](https://data.humdata.org/dataset/idmc-idp-data-ssd) | `data/raw/idp/` | 15 annual + 158 disaster events (no coordinates) |
| Displacement sites | [HDX — IOM DTM Site Assessment](https://data.humdata.org/dataset/south-sudan-displacement-data-site-assessment-iom-dtm) | `data/raw/displacement_sites/` | 77 sites in Round 11 (100% GPS coverage) |

Raw data is **not committed to git** (see `.gitignore`). Re-download with `./scripts/download_datasets.sh`.

### Phase 1 key findings

1. **Two independent road networks** exist: a small humanitarian basemap (976 features, includes rivers) and a much larger HOT OSM export (20,512 filtered road segments). They share no segment identifier.
2. **Health facilities** come in two overlapping files with different schemas. SS 2023 has facility type and codes; WHO 2025 has more rows and better coordinate coverage.
3. **IDMC IDP data** provides national/disaster aggregates only — no georeferenced camp locations.
4. **IOM DTM site assessments** provide georeferenced displacement sites with admin P-codes and per-site IDP counts. This fills the camp-location gap identified during early exploration.
5. **HOT OSM filter applied:** only `highway` ∈ {`primary`, `secondary`, `tertiary`, `unclassified`} retained (saved to `data/raw/roads_hotosm/filtered/roads_lines_filtered.gpkg`).

Full analysis: `docs/phase1_data_understanding.md`

### Bootstrap on a new machine

**Windows 10 + Miniforge (recommended):**

```powershell
.\scripts\setup.ps1
conda activate dm-south-sudan
python scripts\download_datasets.py
python scripts\explore_datasets.py
python scripts\visualize_data_validation.py
```

**macOS / Linux (venv):**

```bash
./scripts/setup.sh
./scripts/download_datasets.sh
data_env/bin/python scripts/explore_datasets.py
data_env/bin/python scripts/visualize_data_validation.py
```

**Conda on any OS:**

```bash
conda env create -f environment.yml
conda activate dm-south-sudan
python scripts/create_dirs.py
python scripts/download_datasets.py
```

GeoPandas/GDAL on Windows should be installed via **conda-forge** (`environment.yml`), not pip alone.
